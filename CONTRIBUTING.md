# Contributing to OpenClaw Setup Guide

Thanks for your interest in improving this guide. Contributions from the legal tech community are welcome.

## How to Contribute

### Reporting Issues

- Open a GitHub issue describing the problem
- Include your OpenClaw version, OS, and Node.js version
- For security issues, **do not** open a public issue — email the maintainer directly

### Suggesting Improvements

- Open an issue with the `enhancement` label
- Describe what's missing or could be better
- Include your use case (firm size, practice area, setup)

### Submitting Changes

1. Fork the repository
2. Create a feature branch: `git checkout -b improve/section-name`
3. Make your changes
4. Test any configuration examples you modify
5. Submit a pull request with a clear description

## What We're Looking For

- **Corrections** — Fix errors in configs, commands, or procedures
- **Additional platforms** — Guides for CentOS/RHEL, ARM, etc.
- **New integrations** — MCP tool examples for other legal software
- **Security improvements** — Better hardening recommendations
- **Cost optimization tips** — Cheaper model configs that work well
- **Troubleshooting additions** — Common issues and their fixes

## Guidelines

- Keep it practical — working examples over theory
- No real API keys, tokens, or credentials in examples
- Use placeholder format: `<YOUR_TOKEN_HERE>` or `sk-xxxxxxxxxxxxx`
- Test your config changes before submitting
- Follow the existing formatting style (markdown, code blocks, tables)
- One logical change per pull request

## Code of Conduct

Be professional. This is a guide used by law firms — keep contributions clear, accurate, and respectful.
